MSAL Redirect Race Condition Repro

Simulates cross-app MSAL interference: a host app doing a redirect flow while a guest component tries to auth silently.

Guest App Config

Prod flow config — used by "Run Prod Flow" and by S1 scenarios. Mirrors get-ad-user-account.ts.

SID override: forces sid=<value> on silent flows. Garbage value reproduces AADSTS160021.
loginHint override: forces login_hint=<value>. When set, MSAL skips the account lookup entirely (no sid extracted). This is the proposed fix for Mode 2.

Scenarios

Each scenario uses the full state machine (may redirect through AD or self) and records an outcome to the results table below.

S0: Rapid-abort banning of guest ssoSilent (Mode 1)

Fires guest ssoSilent and immediately redirects the page to self, aborting the in-flight iframe. Repeats N times, then a final attempt is allowed to complete. Tests whether AD's loop-detection throttling trips after repeated aborted silent flows — the failure mode SSO_SILENT_DELAY_MS is designed to mitigate.


Respects the guest's SSO_SILENT_DELAY_MS on the final attempt so you can A/B the mitigation. Set it to 0 to see the raw failure; set it to e.g. 2000 to verify the fix.

S1: Host loginRedirect → (timing) → Guest Prod Flow (Mode 2)

Host does a real interactive redirect. On return, the guest's production flow (acquireTokenSilent → ssoSilent → loginPopup) runs — but when it runs relative to the host's handleRedirectPromise is configurable, because production has these two racing in parallel on page boot.


S2: Abandoned silent flow diagnostic (guest only, no host)

Fires a guest ssoSilent, aborts it mid-flight by redirecting to self, then fires one more attempt and reports the result. After S2 completes, manually click Guest: ssoSilent to test the third attempt yourself. Captures storage snapshots and diffs to isolate client-side vs server-side effects.


Flow: abort → attempt 2 (automated) → you click Guest: ssoSilent (attempt 3).
What it captures: storage snapshots pre-abort, post-abort, and post-attempt-2 with diffs between each. For server-side diagnosis, capture a HAR while running.

Timing modes: after = host finishes processing its redirect, then guest runs (sequential, with optional gap between). parallel = host handleRedirectPromise and guest flow start concurrently via Promise.all — closest to production boot. before = guest flow runs first against in-flight redirect state (URL has ?code=..., shared sessionStorage still dirty), then host handleRedirectPromise cleans up afterwards. The gap input inserts a delay between the two actions in "after" and "before" modes; in "parallel" mode it's ignored.

S3: Guest ssoSilent abort → host loginRedirect → guest ssoSilent retry

Fires a guest ssoSilent, aborts it mid-flight (redirect to self), then on return the host does a full-page loginRedirect (simulating Polaris's AD flow), and on return from that the guest retries ssoSilent. Tests the full production sequence: our silent flow gets abandoned by a host-app redirect, the host completes its own AD flow, then we retry.
